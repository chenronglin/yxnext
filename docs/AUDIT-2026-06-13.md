## 二、严重问题（建议立即修复）

### S1. 作者可越权读取任意作者的预发记录（IDOR）
- 位置：`server/modules/si/si.service.ts:743-768`（关键在 759 行）；入口 `app/api/si-prepublish/route.ts:17`
- 问题：`listSiPreissues` 的 where 中，角色限定先展开、请求参数后展开：作者请求 `GET /api/si-prepublish?authorId=<他人ID>` 时，`authorId: actor.userId` 被覆盖为任意作者 ID，可读到他人全部非收回预发记录（含 `siSnapshotJson` 中的选题标题、Trope、Fresh Twist、核心梗概）。违反 BUS §7「作者仅可见预发给自己的」。
- 方案：`...(actor.role !== "author" && authorId ? { authorId } : {})`，并补作者带 `authorId` 参数的越权测试。

### S2. `active_pair_key` 语义错误，「一个作者只绑一个编辑」没有数据库兜底
- 位置：`server/shared/invariant-keys.ts:9-13`、`prisma/schema.prisma:250`、`server/modules/admin/admin.service.ts:1753-1785`
- 问题：键值为 `"editorId:authorId"`，作者 5 绑编辑 1（`1:5`）与绑编辑 2（`2:5`）互不冲突——唯一索引只能防「同一对重复绑定」，防不了「一作者多编辑」。并发创建两个不同编辑对同一作者的绑定时，事务内 `findFirst` 先查后写双双通过，产生双活动绑定；而预发、归属调整等授权全部建立在绑定唯一之上。
- 方案：`active_pair_key` 改为仅 `String(authorId)`（活动时写、解绑置 null）；清洗存量数据后迁移；修正注释。

### S3. 修订模式下 Delete（前向删除）键完全卡死【已实证】
- 位置：`components/doc/tiptap/extensions.tsx:971-985`（handleKeyDown）、`:439-440`（markDeletedRange 光标回置）
- 问题：mark-only 删除后把光标设回 `range.from`（纯 addMark 事务 mapping 为恒等映射），Delete 的目标永远是同一个已标记字符。实测连按 4 次 Delete 光标始终停在原位，只有第一个字符被标记。编辑在修订模式下前向删除事实上不可用。
- 方案：Delete 时光标移到被标记区段之后（`resolvedTo`）；targetRange 计算时跳过紧邻的已 deleted 文本（对齐 Word 行为：越过删除痕迹）。

### S4. 没有「接受/拒绝修订」机制；唯一兜底操作会产生脏文本
- 位置：`components/doc/tiptap/discussion-sidebar.tsx:261-285`；全仓无 accept/reject 实现
- 问题：修订生命周期只有「产生」没有「消解」。侧栏唯一的「删除标记」按钮对三种修订的后果各不相同且 replace 直接出错：文档中 replace 是 `[inserted 新文][original 原文]` 相邻共存，剥掉标记后两段同时变正文，产生「新文原文」脏稿；「接受删除」则完全无途径。审批流（`doc.service.ts approveDoc`）直接把带标记的 contentJson 定稿。
- 方案：实现 `acceptRevision(id)` / `rejectRevision(id)`（accept：删除 deleted/original 文本、剥 inserted 标记；reject：删除 inserted 文本、剥 deleted/original 标记），区段从右向左应用或经 `tr.mapping` 映射；侧栏改为接受/拒绝双操作；提交/审批前对「存在未处理修订」给出拦截或一键全部处理。

### S5. 选区含已有 inserted 修订时直接输入，插入文本被误标为 original【已实证】
- 位置：`components/doc/tiptap/extensions.tsx:326-353`（applyInsertedText 替换分支）、`:574-586`（finalizeComposition 同样问题）
- 问题：replace 路径对整个原选区无差别 addMark original，选区内别人/自己刚插入的文本（从未存在于底稿）进入 original 轨道：plainText/版本审计把它当原文；一旦走「删除标记」它会凭空回到正文。
- 方案：replace 分支按 `markDeletedRange` 的方式分段处理——inserted 段直接删除，只有真实原文标 original；IME 路径缓存 slice 时剔除 inserted 段。

### S6. 结构性编辑完全绕过修订追踪
- 位置：`components/doc/tiptap/extensions.tsx:951-990`（插件只挂了 handleTextInput/handlePaste/handleKeyDown(Backspace|Delete)/composition 四个入口）
- 问题：选中文本按 Enter（真删+拆段）、Cmd/Ctrl+X 剪切、编辑器内拖拽、段首 Backspace 合并段落、仅 text/html 的剪贴板粘贴——全部产生**无标记的真实变更**；段首 Backspace 还会无痕真删紧邻的 editSuggestion 原子块。
- 方案：拦截 Enter（非空选区先走 markDeletedRange）、cut（转标删）、drop（禁用或转标记语义）；段落级结构变化短期直接禁止；长期引入统一事务过滤层（`filterTransaction` 把所有 docChanged 步骤改写为标记语义），而非逐个入口打补丁。

### S7. 移动端（<768px）完全没有导航入口
- 位置：`components/app-sidebar.tsx:47`、`components/app-topbar.tsx:49-55`
- 问题：侧栏 `hidden md:flex`，md 以下整个导航消失；顶栏汉堡按钮只切换 `collapsed` 宽度，不弹抽屉。手机用户登录后被困在当前页。
- 方案：md 以下用 Drawer/Sheet 渲染同一份 `NAV_BY_ROLE`，汉堡按钮按断点分流。

### S8. 账号状态页向所有被驳回用户展示硬编码的虚假「管理员反馈」
- 位置：`app/(auth)/account-status/page.tsx:57-64`
- 问题：`rejected` 状态固定渲染「提交的作品信息不完整……」假文案，与管理员实际填写的驳回原因（审批页必填入库）完全无关。
- 方案：从接口读真实 `rejectedReason` 渲染；接通前先删掉假文案只显示通用提示。

### S9. 「参数管理-SI 主类型」与 SI 表单完全脱节
- 位置：`components/si/si-form.tsx:285`（渲染硬编码 `DEFAULT_MAIN_TYPES`）、`types/si.ts:4-12`、`app/(app)/admin/params/page.tsx:342`
- 问题：管理员在参数页维护主类型（增/停用/排序），页面还提示「停用后不再出现在新建 SI 下拉框」，但 SI 表单和列表筛选根本不请求该接口——管理动作对业务侧零生效。
- 方案：SI 表单与筛选改为请求启用中的主类型；硬编码数组仅作接口失败降级。

### S10. 删除章节无确认弹窗
- 位置：`app/(app)/projects/[id]/chapters/page.tsx:159-177、358-369`
- 问题：点击图标即直接 DELETE，删除按钮紧挨上移/下移按钮，误触即丢稿。全站其他危险操作（收回预发、解绑、禁用用户）都有确认，唯独最不可逆的内容删除没有。
- 方案：补确认 Dialog（展示章节标题与后果说明）。

### S11. 会话过期/越权无全局处理；编辑侧路由缺服务端守卫
- 位置：`lib/api.ts:16-30`；`app/(app)/si/`、`app/(app)/review/page.tsx:78-80`
- 问题：401 后用户停留在页面，每次操作只见红色 banner，无跳转登录引导；作者直接访问 `/si`、`/si/new`、`/review` 能渲染完整编辑侧 UI（填完整张表提交时才报错）——对比 `admin/layout.tsx`、`governance/layout.tsx` 都有服务端 redirect。
- 方案：`fetchJson` 统一拦截 401 → 跳登录；为 `(app)/si`、`/review` 补与 admin 一致的服务端 layout 守卫。

### S12. `ignoreBuildErrors: true`——类型错误被构建静默吞掉
- 位置：`next.config.mjs:3-5`
- 问题：TS 错误不阻断构建，类型安全形同虚设，配合大量 BigInt/字符串转换场景风险高。
- 方案：移除该配置，修完存量类型错误；CI 加 `tsc --noEmit` 闸门。

## 三、中等问题

### A. 后端业务逻辑

1. **chapter 阶段计划永不 completed → 系统性逾期误报**
   `server/modules/doc/doc.service.ts:1108-1170`（advanceProjectAfterApprove 无 chapter 分支）、`project.service.ts:1293-1303`（unlockProjectQc 不 complete chapter）。质检解锁后 chapter 计划仍 `unlocked`，被持续推成 `due_soon/overdue`，发出错误预警通知/待办，污染项目列表 overdue 标记与看板统计。方案：unlockProjectQc 同时 `completeStagePlan(tx, projectId, "chapter", now)` + 存量数据回填。

2. **项目 complete/archive/cancel 不清理打开的待办；统计不过滤生命周期**
   `admin.service.ts:2651-2682`、`project.service.ts:1348-1414`（无 todo 清理）；`admin.service.ts:867-875`、`workbench.service.ts:637-656`（overdue 统计不过滤 lifecycleStatus）。已完成/取消项目的逾期值被冻结并永久计入「逾期项目」。方案：生命周期转移事务内批量 cancel 打开待办；统计 where 增加 `project: { lifecycleStatus: "active" }`。

3. **`syncActiveProjectTimelineStatuses` 在读路径全量扫描 + 非事务 + 并发重复通知**
   `server/shared/project-stage-timeline.ts:188-288`，被 17 处 GET 调用。每个读请求加载全系统活动项目的全部阶段计划并逐条 update；状态翻转「读旧值→写新值」并发下重复发 stage_warning 通知（无去重键）；todo upsert 并发可抛 P2002 使 GET 变 500。方案：移出读路径改定时任务；用 `updateMany({ where: { timelineStatus: 旧值 } })` 条件翻转，affected=1 才发通知。

4. **cleanText 校验被回退顺序架空（M7 仅形式修复）**
   `doc.service.ts:455`（normalize 先把缺省 cleanText 回退为 plainText）→ `:1267-1276`（再校验）。省略字段即绕过，带修订/批注的 plainText 原样进入 cleanText 并冻结进 Revision、流入导出。方案：用原始 `input.cleanText` 校验；submit 封存前复检；更彻底的方案见 B-9（服务端重导投影）。

5. **`transitionGovernanceProject` 缺生命周期状态机**
   `admin.service.ts:2611-2821`。completed 可重复 complete、archived/cancelled 可互转覆盖时间戳、active 项目可被 restore（无差别重建活动稿，可能搅乱流转中的 holder/draft）。方案：状态转移表 + 非法转移 409。

6. **`convertSiPreissueToProject` 三处缺口**
   `si.service.ts:1359-1620`。①无 P2002 翻译，并发转项目冒 500；②`sourceSiId` 无唯一约束，同一 SI 预发给两作者并发转换 → 两个项目；③不校验编辑-作者绑定仍 active（与归属调整口径不一致）；④转项目后同 SI 其他 `preissued` 记录永久悬挂。方案：catch 翻译 + 唯一约束/条件抢占 + 绑定校验 + 事务内批量 recall 并通知。

7. **`updateManagedUser` 角色变更无护栏，可绕过 M4 清空管理员**
   `admin.service.ts:1222-1247`。可把最后一个 active 管理员改成 author；把有活动绑定/在管项目的用户互转 editor/author 立刻破坏可见性模型。方案：admin→非 admin 复用「最后管理员」检查并禁止自改；editor/author 互转前检查活动绑定、在管项目、活动预发。

8. **注册/用户治理/改邮箱先查后写、无 P2002 翻译**
   `auth.service.ts:236-269`、`admin.service.ts:1102-1117、1202-1220`、`account.service.ts:148-168`。并发重复注册/改邮箱 → 500（注册是公网入口）。方案：四处统一 catch + 翻译为既有业务码 409。

9. **可删除带活动预发的 SI，级联硬删作者侧记录且无通知**
   `si.service.ts:1896-1977`、`schema.prisma:367`（Cascade）。作者「我的 SI」条目无声消失。方案：删除前要求先收回，或事务内批量 recall + 通知再删。

### B. 编辑器与协作

1. **无未保存离开保护；冲突暂停后继续输入必丢**
   `components/doc/doc-editor.tsx:152-158、247-260`。dirty 时关页无提示；`DOC_LOCK_VERSION_CONFLICT` 后编辑器未置只读，用户可继续打字但内容不进 latestPayloadRef，提示却让用户刷新（照做即丢失）。方案：beforeunload + 路由守卫；冲突后置只读 + 「复制当前内容」逃生门。

2. **自动保存失败后 900ms 无限重试，无退避无上限**
   `doc-editor.tsx:162-175`。确定性错误（400 校验失败）也每秒打一次接口。方案：指数退避 + 最大次数，转手动重试 UI。

   `doc-editor.tsx:262-274` 与 `:171-174`。被 await 的保存在 finally 里重新排队 debounce，无人再 cancel，遗留定时器与 submit 并发 → save 先落库则 submit 必 409。方案：await 后再次 cancel；runLatestSave 增加 dirty 检查。

5. **跨段落选区一次标删拆成 N 条独立修订**（`extensions.tsx:409-433`，块边界使邻接判断失败）。方案：单次调用统一 id/groupId（循环外生成）。

6. **只读状态下侧栏/编辑建议卡片仍可改文档（幻影编辑）**
   `discussion-sidebar.tsx:435-447`、`extensions.tsx:1043-1150`。`editor.view.dispatch` 不受 editable 限制，非持有人操作本地生效但不落库，刷新即回滚；也无「不能删他人批注」校验。方案：按钮以 `editor.isEditable` 渲染/禁用；按 createdBy 限制删除权。

7. **作者修改返稿时，在编辑的 inserted 修订内部输入会被记到编辑名下**
   `extensions.tsx:951-956` + RevisionMark `inclusive:false`。追踪关闭时默认 mark 继承使新字符披上编辑的 revision mark。方案：追踪关闭时也注册 handleTextInput 过滤继承的 revision mark。

8. **历史版本「工作视图」渲染 plainText，三种角色文本无差别拼接**
   `doc-version-detail.tsx:74-80、141-155`。审计场景不可读。方案：只读 Tiptap 渲染 contentJson（带标记样式）。

9. **后端完全信任前端投影**
   `doc.service.ts:438-493`。cleanText/wordCount/counts 全程用客户端值，而 `deriveNovelDocProjection`（lib/novel-doc.ts，纯函数无 server-only 依赖）本可服务端重导。持有人篡改 payload 可让定稿正文与 contentJson 脱钩。方案：保存/封存时服务端重导并覆盖（或比对不一致 400）——此方案同时根治 A-4。

10. **每次击键约 4~5 次全文档级 O(n) 操作，其中一次结果被丢弃**
    `novel-tiptap-editor.tsx:245-249`（onUpdate 派生投影→`doc-editor.tsx:247` 以 `_nextProjection` 弃用）、`:262-276`（双 JSON.stringify 同步比较）、`extensions.tsx:724-755`（BlockId 全文扫描）、`:1263-1291`（讨论高亮全量重扫不用 DecorationSet.map）。数万字章节有可感输入延迟。方案：投影延迟到防抖保存时计算；版本计数替代 JSON 串比较；BlockId 只扫事务影响范围；DecorationSet 增量 map。

11. **批注随文本删除湮灭，无孤儿处理**
    `extensions.tsx:758-813`（body 全存 mark attrs）。作者一次退格可无声抹掉编辑的长批注。方案：批注实体外置 + mark 只存锚点；或删除含批注文本时确认 + 失锚批注列表。

7. **永久 disabled 占位按钮**：`si/page.tsx:210-224` 每行渲染禁用的「归档/删除」且无提示；`chapters/page.tsx:349-357` 禁用图标含义不明。
8. **报表「统计维度」下拉无效**：`reports/page.tsx:63-74` dimension state 从未被读；作者看板 hint 描述了不存在的时间筛选。
9. **`/admin/audit` 是导航孤儿页**：`config/navigation.ts:27-39` adminNav 无入口。
10. **表单客户端校验薄弱**：新增用户可空提交、改密不校验两次一致、注册无邮箱格式校验（项目已有 zod 可前端复用）。

### D. 安全与工程

1. **注册/忘记密码无限流**：`app/api/auth/register/route.ts`、`forgot-password/route.ts` 未接 rate-limit（login 已有）。公网可批量注册垃圾账号、刷忘记密码通知。方案：复用现有限流器，按 IP 限注册频率。
2. **seed 弱密码 + 无生产防护**：`scripts/seed.ts:21-64` 使用 `admin00000` 等弱口令，无 NODE_ENV 检查。生产误执行即植入弱口令账号。方案：从环境变量读密码或随机生成打印一次；`NODE_ENV === "production"` 时要求显式 `--force`。
3. **SESSION_SECRET 与 jose 是死配置/死依赖**：`.env`、`.env.example` 都有 `SESSION_SECRET`，但代码 0 处引用（会话是数据库不透明 token，无签名需求）；`jose` 在 package.json 却无 import。误导运维做无意义的密钥轮换。方案：删除两者，或注明保留原因。
4. **临时密码熵偏弱**：`admin.service.ts:1429` `randomBytes(5).toString("hex")` = 40 bit。虽有强制改密兜底，建议 ≥ 8 字节。
5. **会话无滑动续期、过期记录无清理**：`session.ts:14` 固定 7 天；`user_sessions` 过期/撤销记录永久堆积无清理任务。方案：活跃续期（可选）+ 定期清理任务（保留审计需要的时长）。
6. **无 middleware、无 security headers**：路由保护完全依赖每个 layout/route 自觉调用（当前覆盖已完整，但无纵深防御层）；`next.config.mjs` 无 CSP/X-Frame-Options/HSTS 等。方案：补 middleware 做粗粒度登录态拦截 + headers 配置。CSRF 当前依赖 SameSite=lax（基本可接受），建议补 Origin 校验做纵深。
7. **分页覆盖率低**：服务层 32 处 `findMany` 仅 7 处 `take`。`listManagedUsers`、通知、绑定等全量返回，数据增长后接口越来越重。
8. **编辑器测试跑在「零插件」编辑器上（测试基建失效）**：`vitest.config.ts` `environment: "node"`，tiptap v3 只在 mount（需要 DOM）时注册 plugins——实测 `editor.state.plugins` 为空数组。现有 revision 测试只覆盖了纯函数，`handleTextInput/handleKeyDown`、plugin state、undo 交互**从未被执行**，S3 这类一跑就现形的 bug 因此存活。方案：装 jsdom，编辑器集成测试加 `// @vitest-environment jsdom`，并断言 `plugins.length > 0` 作防回归哨兵。
9. **`tsconfig.tsbuildinfo`、`next-env.d.ts` 已入库**：构建产物反复制造 git 噪音（当前工作区也是这两个文件 dirty）。方案：`git rm --cached` + 加入 .gitignore。

## 五、架构优化建议

1. **编辑器引入统一事务过滤层**：当前逐 props 入口打补丁的方式注定漏（S6 即证据）。用 `filterTransaction`/`appendTransaction` 把修订模式下所有 docChanged 步骤统一改写为标记语义，一处管控全部输入路径。
2. **投影计算「保存时 + 服务端」双置**：`deriveNovelDocProjection` 是纯函数，移到防抖保存时计算（解性能）+ 服务端重导覆盖（解信任），「真相源是 contentJson」才名副其实。
3. **统一数据获取层（SWR/TanStack Query）**：~19 个页面手写 loading/message/cancelled 三件套且有遗漏，统一后免费获得竞态消除、缓存、焦点重验证；`fetchJson` 升级为带 401/403 拦截的客户端。
4. **首屏只读列表迁回 Server Component**：项目/SI/待办/通知列表完全可像 `si/[id]/page.tsx` 一样服务端直出（直接调 service），消除客户端瀑布。
5. **全局反馈体系**：sonner toast + 统一 `<ConfirmDialog>`，废除 31 处 banner 复制和三套并存的确认方式，确立「危险操作必确认、成功必 toast」规范。
6. **筛选/分页 URL 化**：统一 searchParams 读写（修复深链失效），增长型列表加服务端分页。
7. **逾期同步改定时任务**：从 17 个读路径移除 `syncActiveProjectTimelineStatuses`，改 cron + 条件翻转，通知以翻转成功为前提。
8. **共享枚举映射层**：`rejected→returned`、`release→质检`、状态 label/tone 收敛到 `server/shared`/`lib` 单一来源，前后端共用。
9. **测试基建**：jsdom 环境跑编辑器集成测试；补「任意编辑序列后 cleanText(contentJson) === 接受全部修订后的 plainText」性质测试；补越权（S1 类）、并发（P2002）路径测试。
10. **工程口径**：去掉 `ignoreBuildErrors`、清理死依赖（jose）/死配置（SESSION_SECRET）/死代码、tsbuildinfo 出库、CI 加 `tsc --noEmit` + `pnpm test`。

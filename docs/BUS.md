# 阅享小说协作管理与审稿交流平台 · 业务说明书（BUS）

> 本文是平台的业务首文档，描述业务对象、角色职责、核心流程、协作与审核规则、权限边界与验收要点。
> 本文不是接口文档或数据库设计文档；凡涉及存储与渲染的实现细节，以"文档 JSON / 前端负责"一句带过。
> 项目四阶段页面文案统一为：`梗概 / 细纲 / 正文 / 质检`；系统内部阶段编码固定为 `synopsis / outline / chapter / release`。

---

## 1. 产品定位与核心模型

### 1.1 产品定位

阅享是小说从选题创意到成稿交付的协作管理与审稿交流平台，定位为**选题策划库 + 审稿沟通室 + 项目集散地**。作者的日常写作在 Word、Scrivener 等外部工具完成；平台只承接"已准备提交评审的阶段产出"，负责内容提交、审稿沟通、版本留档与项目流转。

### 1.2 四阶段与统一 Doc

项目核心四阶段固定为：`梗概 → 细纲 → 正文 → 质检`。四类阶段产出统一抽象为 **Doc（文档）**，在同一套"通用 Doc 编辑器"中提交、评审、修正：

- 每个 Doc 以**一份完整的文档 JSON（结构化富文本）**承载正文与全部文内协作信息（修订标记、批注、编辑建议）。
- 后端**只整体存取这份 JSON，不解析其内部节点结构**；节点的渲染、隐藏与编辑权限完全由前端负责。
- 后端可保存字数、纯文本、导出文本、摘要等派生字段，仅用于列表/搜索/统计/导出，**不作为编辑真相源**；字数优先由前端在导入、粘贴、提交时计算并提交。
- 新流程中的**干净稿 / Clean 正文**不是一份单独的作者原稿，而是从当前最新稿件或指定 Revision 中计算出的原文视图：去除修订标记、批注高亮、编辑建议卡片等协作标记后得到的正文内容。

### 1.3 共创接力式评审（非实时协同）

Doc 沟通采用**共创接力式评审**，作者与编辑轮流持有同一份最新可编辑稿件；同一时刻只有一个角色可编辑正文，从机制上避免多人同改造成的冲突与版本混乱：

```
作者编辑/保存 → 作者提交 → 编辑获得编辑权并可修改/批注
        → 编辑退回 → 作者获得编辑权，在编辑退回后的内容上继续修改
        → 作者再次提交 → 编辑再次修改/批注 → …… → 编辑通过，当前 Doc 成为定稿/阶段有效内容
```

每次**提交 / 退回 / 通过**都会生成一份不可修改的历史快照（Revision）；系统中始终只有一份当前工作稿（CurrentDraft）作为最新可编辑稿件，前端编辑器默认加载它进行修改。

### 1.4 当前工作稿 + 历史快照模型

这是 Doc 协作的核心设计，务必准确理解：

1. **CurrentDraft 是唯一当前可编辑稿件**。每个 Doc 同一时刻最多只有一份 `active` CurrentDraft，记录当前持有编辑权的角色、用户、基于哪个历史版本继续编辑、完整文档 JSON 与乐观锁版本号。
2. **Revision 是每次交接时冻结的历史快照**。作者提交、编辑退回、编辑通过都会封存当前工作稿并生成 Revision；Revision 永久只读，用于历史回溯与审计。
3. **作者和编辑共创同一条内容主线**。作者提交后，编辑在最新稿件上修改正文、添加批注/编辑建议；编辑退回后，作者在编辑退回后的内容上继续修改，而不是回到另一份原稿重新写。
4. **单一编辑权由状态与持有人共同决定**。`holder_role` / `owner_role` 决定当前谁能编辑；非持有人只能查看当前内容或历史记录，不能保存正文变更。
5. **当前持有人可多次保存**。在提交、退回或通过之前，当前持有人可以多次保存草稿；保存只更新 CurrentDraft，不生成 Revision。只有交接动作才生成历史快照。
6. **审核通过时定稿来自最后一次通过生成的 Revision**。通过后项目或阶段进入有效状态，CurrentDraft 封存，`final_revision_id` 指向最终快照；导出或阅读时展示的是该最终快照去除修订标记后的 Clean 正文。

### 1.5 项目范围

1. SI 是独立的选题策划库，不属于项目四阶段。
2. 项目只包含梗概、细纲、正文、质检四个执行阶段，全部产出统一进入 Doc。
3. 文档 JSON 结构由前端负责；后端保存当前工作稿、历史 Revision、最终 Revision、完整 JSON 与必要派生字段。
4. 作者端聚焦：内容导入、草稿暂存、提交审核、查看历史版本、在退回稿上继续修改。
5. 编辑端聚焦：SI 管理、项目推进、审稿共创（修改正文、点评/批注/编辑建议）、退回/通过处理。
6. 管理员以治理视角为主：用户、绑定关系、参数、项目治理、统计、终稿下载。
7. AI 作为模块入口保留，围绕审稿与智能辅助扩展。

## 2. 角色与职责

### 2.1 管理员

平台治理者，不承担日常审稿。**可做**：全局统计看板；用户增改/禁用/启用、注册审批、忘记密码人工重置、重置密码；维护编辑-作者绑定；维护 SI 主类型等参数；全局项目列表；调整项目归属；设置四阶段计划时长；标记项目完成/归档/取消/恢复；下载终稿。

### 2.2 编辑

选题策划、项目推进与内容审核者。**可做**：创建/修改 SI；将 SI 预发给已绑定作者；查看预发列表；从预发记录确认转项目；收回预发中的 SI；删除未转项目的 SI；查看自己负责的项目并进入任意可见 Doc；审阅四类 Doc 并执行通过/退回；在自己持有编辑权时修改正文、添加批注（含删除/替换/增加预设批注）、加编辑建议；满足条件后手动解锁质检；导出项目；质检通过后标记项目完成。**不可**：在作者持有编辑权时改稿；越权修改非自己负责项目；绕过历史快照直接覆盖已封存版本。

### 2.3 作者

承接选题、把外部完成的阶段产出导入系统接受评审。**可做**：在"我的 SI"查看预发给自己的 SI 及说明/状态；SI 转项目后进入项目；在自己持有编辑权时通过通用 Doc 编辑器导入/粘贴/修正四类阶段内容；保存草稿；可填写提交说明并提交审核；编辑退回后在编辑修改后的内容上继续修改；查看审核结果、退回意见与历史版本。**不可**：修改/收回/删除 SI、决定 SI 是否转项目；审核 Doc；管理项目归属；管理系统参数；在编辑持有编辑权时改稿；修改已封存历史快照。

### 2.4 外部注册用户

可从登录页提交注册申请。注册入口只接受邮箱，不接受手机号；申请提交后默认 `待审批`，管理员审批通过前不可进入业务流程。忘记密码入口也只接受邮箱，系统只通知管理员人工处理，不向用户发送重置链接。

## 3. 业务对象

### 3.1 SI（创意设定 / 选题）

独立于项目执行流的选题资源，含标题、主类型、Trope、适配作者、备注、Fresh Twist、核心故事梗概、创建编辑、状态。Trope 由用户在 SI 中直接输入，不作为后台统一配置参数。规则：不属于四阶段；可预发给多个作者；必须从某条预发记录确认转项目；作者只能看预发给自己的 SI；已转项目的 SI 不可删除。

SI 由所属编辑维护版本历史：编辑创建或保存 SI 草稿时系统自动留存一份版本快照，编辑可查看自己负责 SI 的历史版本并回退到指定版本；回退会把历史内容写回 SI 主记录，并生成一条新的当前版本快照。SI 版本仅服务于选题策划追溯，不属于项目四阶段 Doc 版本。

### 3.2 SI 预发记录

表示"某 SI 预发给某作者"，含 SI、编辑、作者、预发说明、时间、状态、（若已转）项目、预发时 SI 内容快照。规则：同一 SI 对同一作者只保留一条有效记录；已收回记录在作者端隐藏；已转项目记录不可收回、不可重复建项目。

### 3.3 项目

正式协作的业务容器，含来源 SI、标题、简介、负责编辑、负责作者、生命周期状态、当前阶段、阶段计划、关联 Doc、Release 状态及通知/审核/导出/统计等数据。规则：一项目绑定一编辑一作者；创建后默认进入梗概阶段；阶段推进由 Doc 审核结果与 Release 状态驱动；完成后默认只读，需继续协作须由管理员恢复。

数据模型口径：新流程中的 `status`、`holder_role`、`active_draft_id`、`latest_revision_id`、`final_revision_id` 属于"稿件协作单元"。在本平台四阶段体系中，该协作单元对应 Doc；Project 继续作为业务容器汇总多个 Doc，正文阶段仍允许不同章节 Doc 并行处于不同接力状态。

### 3.4 Doc（文档）

系统统一的内容协作对象，四类：`梗概 Doc`（一项目一份）、`细纲 Doc`（一项目一份）、`正文 Doc`（多份，每份一章节）、`质检 Doc`（一项目一份，对应 `release`）。含所属项目、类型、标题、排序值（正文用）、状态、当前持有人、当前工作稿、最新历史快照、最终历史快照、字数、摘要、最近交接说明（可选）/提交人/审核人/操作时间。数据规则见 §1.2 与 §5.3。若按字段表达，Doc 至少维护 `status`、`holder_role`、`active_draft_id`、`latest_revision_id`、`final_revision_id`。

### 3.5 CurrentDraft（当前工作稿）

CurrentDraft 是 Doc 中当前唯一可编辑的内容，含 Doc、当前编辑者角色（作者/编辑）、当前编辑者用户、基于哪个 Revision 继续编辑、完整文档 JSON、状态（active/sealed/archived）、乐观锁版本号、创建/更新时间。规则：同一 Doc 同一时刻最多只有一份 `active` CurrentDraft；当前持有人可在交接前多次保存；提交、退回、通过时当前工作稿被封存并进入历史链。

### 3.6 Revision（历史快照）

Revision 是每次交接后冻结的完整稿件快照，含 Doc、版本号（如 R1/R2/R3）、基于的上一版本、完整文档 JSON、动作（作者提交/编辑退回/编辑通过）、操作角色、操作人、提交或审核说明（可选）、创建时间、内容哈希（可选）。规则：每次提交/退回/通过都生成 Revision；Revision 永久不可修改；`base_revision_id` 构成版本链；审核通过时最终 Revision 成为该 Doc 的定稿/阶段有效内容。

### 3.7 批注

编辑对选中文本的**单向点评**，作为文档 JSON 内部的包裹节点随文保存，含被点评文本、批注内容、创建人、时间。规则：由前端编辑器生成维护，与文本物理绑定、随文移动；后端不单独建立锚点/回复串/解决状态；编辑持有编辑权时可增改删；作者可查看但不回复、不解决；阅读模式隐藏批注高亮与侧栏。

### 3.8 编辑建议卡片

编辑在段落旁的**卡片化协作建议**（结构/逻辑/节奏/表达方向），作为文档 JSON 内部节点保存，不属于正文文本。规则：编辑持有编辑权时可增改删，作者可查看但不直接编辑建议卡片；不可被划词批注；阅读模式完全隐藏；由前端负责渲染与权限。

### 3.9 Release / 质检

发布阶段，业务页面统一显示为"质检"。规则：使用一份质检 Doc；解锁时初始内容来自全部已通过正文 Doc；解锁后修改只作用于本 Doc、不回写单章；审核通过后项目具备完成条件；项目终稿导出优先以质检 Doc 为真相源。

## 4. 状态字典

**用户**：`正常` / `禁用` / `待审批` / `已驳回`。

**项目生命周期**：`草稿` / `进行中` / `已完成` / `已归档` / `已取消`。

**项目阶段**：`梗概` / `细纲` / `正文` / `质检` / `完成`。

**Doc 状态**（决定当前持有人，见 §6.3）：
- `草稿`：作者持有编辑权，内容已入系统但未提交审核；作者可多次保存 CurrentDraft。
- `已提交待审`：作者已提交并交接给编辑；编辑持有编辑权，可修改正文、批注、添加编辑建议并执行退回/通过。
- `退回待改`：编辑已退回并交接给作者；作者持有编辑权，在编辑退回后的内容上继续修改。
- `审核通过`：当前内容已通过，最终 Revision 成为该阶段有效内容；默认无人持有编辑权。
- 规则：只有 `已提交待审` 可执行退回/通过；每个可编辑状态都必须有且只有一份 `active` CurrentDraft；已通过 Doc 不再支持重新开启作者编辑。

**编辑权持有人**：`author` / `editor` / `none`。业务状态与持有人必须一致：`草稿`、`退回待改` 对应 `author`；`已提交待审` 对应 `editor`；`审核通过` 对应 `none`。

**SI**：`草稿` / `预发中` / `已转项目` / `已归档`。
**SI 预发记录**：`预发中` / `已确认转项目` / `已收回`。
**阶段计划**：`未开始` / `进行中` / `即将到期` / `已逾期` / `已完成`。

## 5. 核心业务流程

### 5.1 从 SI 到项目

编辑创建 SI 草稿 → 填写标题/主类型/Trope/适配作者/备注/Fresh Twist/核心故事梗概 → 预发给已绑定作者（作者在"我的 SI"可见）→ 编辑在预发列表选某条记录"确认转项目" → 系统创建项目并绑定来源 SI、编辑、作者 → 项目进入梗概阶段并准备梗概 Doc → 作者从"我的 SI"或"我的项目"进入项目。

### 5.2 四阶段推进与门禁

| 阶段 | 解锁条件 | 计时起点 |
|---|---|---|
| 梗概 | 项目创建即进入 | 确认转项目 |
| 细纲 | 梗概 Doc 审核通过 | 梗概通过 |
| 正文 | 细纲 Doc 审核通过 | 细纲通过 |
| 质检 | 全部正文 Doc 通过后，**编辑手动解锁**（不自动） | 手动解锁 |
| 完成 | 质检 Doc 通过后，编辑或管理员标记完成 | — |

门禁补充：未解锁阶段的 Doc 可保存草稿但不可提交审核（如梗概未过时细纲、细纲未过时章节）；质检未解锁前不可保存/提交质检 Doc。正文阶段由多份章节 Doc 组成，各章独立提交/审核、可处于不同状态、可并行推进。

### 5.3 通用 Doc 评审循环（共创接力 + 历史快照 + 单一编辑权）

所有 Doc 共用同一闭环。一轮完整流程：

1. **作者编辑与保存**：作者在外部工具完成内容后导入/粘贴到 Doc，或直接在当前工作稿中修正；前端计算字数随内容保存。此时 Doc 为 `草稿` 或 `退回待改`，`holder_role=author`，作者可多次保存 CurrentDraft，不通知编辑、不推进阶段。
2. **作者提交交接**：作者准备好后可填写提交说明并提交审核。系统封存当前 CurrentDraft，生成 `author_submit` Revision，Doc 进入 `已提交待审`，并基于该 Revision 创建新的 `active` CurrentDraft 交给编辑。
3. **编辑共创审稿**：编辑打开最新 CurrentDraft，在工作模式下可修改正文、添加修订标记、划词批注、整体点评/审核反馈、编辑建议卡片，也可多次保存。作者此时不能编辑正文，只能查看当前内容或历史记录。
4. **编辑退回交接**：编辑认为需要作者继续修改时，填写退回说明并退回。系统封存编辑当前工作稿，生成 `editor_return` Revision，Doc 进入 `退回待改`，并基于该 Revision 创建新的 `active` CurrentDraft 交给作者；作者随后在编辑退回后的内容上继续修改。
5. **编辑通过定稿**：编辑认为内容通过时，执行通过。系统封存编辑当前工作稿，生成 `editor_approve` Revision，设置为最终 Revision，Doc 进入 `审核通过`；按 §5.2 推进下一阶段或满足完成条件。
6. **历史回看**：作者与编辑都可查看提交、退回、通过形成的 Revision 历史链；历史内容只读，不可修改。需要继续协作时只能回到最新 `active` CurrentDraft。
7. 循环 1–6，直到审核通过。

要点：系统中始终只有一份最新可编辑稿件；每次交接都会产生历史快照；Clean 正文始终从最新 CurrentDraft 或指定 Revision 去除修订标记、批注高亮、编辑建议卡片等协作标记后计算得出。

### 5.4 质检解锁与项目完成

全部正文 Doc 通过后，编辑在项目详情点"手动解锁质检"：项目阶段转质检，系统创建/初始化质检 Doc（初始内容来自已通过正文 Doc），通知作者；该 Doc 按 §5.3 循环推进。质检 Doc 通过后，编辑或管理员标记项目完成，项目转 `已完成`（默认只读）并可导出/下载终稿。

## 6. Doc 编辑器与协作规则

### 6.1 当前稿件与历史版本入口（务必区分）

**当前工作稿**与**历史版本**是两个不同入口，严禁混淆：

- 作者侧：
  - **当前稿件**：当 `holder_role=author` 时可编辑最新 CurrentDraft；当编辑持有编辑权或 Doc 已通过时只读。
  - **历史版本**：只读查看每次提交、退回、通过生成的 Revision，可查看对应说明、修订标记、批注与编辑建议，但不可直接修改历史内容。
- 编辑侧：
  - **审稿工作台**：当 `holder_role=editor` 时编辑最新 CurrentDraft，可修改正文、添加修订标记、批注（含删除/替换/增加预设批注）、加编辑建议，并执行保存/通过/退回。
  - **历史版本**：只读查看各 Revision，辅助判断交接顺序与改动脉络。
  - **阅读模式**：以 Clean 正文预览当前稿件或指定历史版本去除协作标记后的效果。

### 6.2 工作模式与阅读模式

- **工作模式**：展示正文 + 修订标记 + 批注高亮 + 批注侧栏 + 编辑建议卡片 + Doc 状态 + 当前持有人 + 提交/退回/通过说明 + 操作按钮。
- **阅读模式（Clean）**：从当前最新稿件或指定 Revision 中去除修订标记、批注高亮、编辑建议卡片等协作标记后展示原文，尽量接近读者最终视角；仅用于预览，不改变保存内容，可随时切回工作模式。

### 6.3 正文编辑权限与状态互斥

同一时刻只有一方动稿，由 **Doc 状态 + holder_role + active_draft_id** 自动互斥：

1. `草稿` / `退回待改`：`holder_role=author`，作者可编辑最新 CurrentDraft；编辑只读查看。
2. `已提交待审`：`holder_role=editor`，编辑可编辑最新 CurrentDraft；作者正文自动只读（明确提示"已提交，等待编辑审核"）。
3. `审核通过`：`holder_role=none`，最终 Revision 默认只读，不再提供重新开启作者编辑。
4. 前端保存必须携带 `lock_version`，后端用乐观锁防止同一持有人多窗口覆盖；保存成功后递增版本号。
5. 提交、退回、通过都会封存当前工作稿并生成 Revision；普通保存只更新 CurrentDraft，不生成历史快照。
6. 保存、提交、退回、通过均记录操作日志。

### 6.4 批注

编辑对选中文本的单向点评：选中后经浮动工具栏添加，工作模式下被批注文本黄色高亮，作为文档 JSON 内部包裹节点随文保存、随文移动。编辑持有编辑权时可增改删；作者可查看但不回复/解决/讨论；不单独保存锚点/回复串/解决状态；阅读模式隐藏。针对正文"增加/删除/替换"的固定格式批注是其特例，见 §6.6。

### 6.5 编辑建议卡片

编辑关联到段落的卡片化建议（大纲/逻辑/结构/节奏/表达方向），作为文档 JSON 内部节点保存，不属于正文原文。编辑持有编辑权时可增改删，作者可查看但不直接编辑建议卡片；不可被划词批注；阅读模式完全隐藏；前端负责渲染与隐藏。

### 6.6 编辑改稿、预设批注与历史快照

编辑持有编辑权时可以直接修改正文，也可以用固定格式工具按钮快速生成批注说明，提高效率与一致性：

1. **直接改稿**：编辑可对正文进行增加、删除、替换；前端可按产品设计保留修订标记，用于工作模式展示改动痕迹。
2. **删除批注**：编辑选中要删除的文本，点工具栏「删除」按钮，可自动在右侧新增一条内容为"此处删除"的批注。
3. **替换批注**：编辑选中要替换的文本，点「替换」按钮，可自动新增一条"此处替换为："批注，编辑可继续在其后补充替换后的内容。
4. **增加批注**：编辑选中一处锚点文本，点「增加」按钮，可自动新增一条"此处增加："批注，编辑可继续在其后补充要新增的内容。
5. 批注与编辑建议随 CurrentDraft 的文档 JSON 保存；退回或通过时完整冻结到 Revision，历史各轮可逐轮回看。
6. Clean 正文不是另一份稿件，而是从最新稿件或指定 Revision 去除修订标记、批注高亮、编辑建议卡片后得到的原文视图。

### 6.7 审核规则

只有 `已提交待审` 且 `holder_role=editor` 的 Doc 可审核；审核人必须是项目编辑或管理员，作者不可审核；审核动作只有通过/退回；退回必须保留反馈；退回和通过都生成 Revision；审核结果通知作者；通过会推进阶段或满足完成条件。

## 7. 权限与可见性

| 维度 | 管理员 | 编辑 | 作者 |
|---|---|---|---|
| 用户/绑定/参数 | 全局管理 | — | — |
| SI | — | 仅自己创建/负责；只能预发给自己绑定的作者 | 仅预发给自己的（只读） |
| 项目 | 全局可见、可调整归属 | 仅自己负责 | 仅分配给自己 |
| Doc 审核 | 可（治理） | 仅自己负责项目的 Doc | 不可 |
| Doc 内容 | 可治理查看，默认不参与日常改稿 | 持有编辑权时改最新 CurrentDraft（正文/修订/批注/建议） | 持有编辑权时改最新 CurrentDraft（正文）；可查看编辑退回内容与历史版本 |
| 质检解锁 | — | 仅自己负责项目 | — |
| 导出/终稿下载 | 终稿下载 | 导出自己负责项目 | — |

调整项目归属后可见性立即按新编辑/作者生效；已完成/归档项目默认只读。

## 8. 阶段计划 / 导出 / 统计

### 8.1 阶段计划与逾期

每个项目固定维护梗概/细纲/正文/质检四个阶段计划，含计划天数、开始/截止/完成时间、预警与逾期状态（计时起点见 §5.2）。阶段即将到期或已逾期时通知相关用户并可跳转关联项目/Doc。阶段计划与逾期状态在首页、待我处理、项目治理、项目列表与详情等处展示。

### 8.2 导出

- 范围：梗概 / 细纲 / 正文合集 / 质检 / 整个项目。
- 口径：各阶段取该 Doc 当前有效 Revision；正文合集取已通过正文 Doc 按章节排序汇总；项目终稿优先取质检 Doc 的最终 Revision；质检未通过时提示终稿尚未完成。导出内容为 Clean 正文，即从对应 Revision 去除修订标记、批注高亮、编辑建议卡片等协作标记后的原文。
- 格式：以 Word 下载为主，保留 Markdown 作为扩展；无有效内容时明确提示。

### 8.3 统计

口径基于项目与 Doc 模型。
- 管理员：用户/编辑/作者数、项目总数与各阶段数、已完成/已逾期项目数、总提交字数、今日提交/审核/退回数、作者提交排行、编辑效率排行。
- 编辑：负责项目数、待审核 Doc 数、退回 Doc 数、即将到期/已逾期项目数、最近处理记录。
- 作者：我的项目数、待修改/待提交 Doc 数、待处理退回稿数、最近提交次数、项目累计字数、最近提交记录。

### 8.4 通知与待办

主要通知与待办口径如下：
- 编辑/作者：围绕 `doc_submitted_for_review`、`doc_returned`、`doc_approved` 三类 Doc 协作事件定义通知与待办。
- 管理员：围绕注册申请、忘记密码等治理事件定义通知；注册申请同时进入管理员待办，忘记密码只发通知不入待办。
- 待我处理只聚合有持久化真相源的任务；待办支持已读/未读，且支持批量标记已读。

## 9. 验收要点

**SI 与项目**：编辑可新建/编辑/预发 SI 并从预发记录确认转项目；作者可查看预发 SI 并在转项目后进入；预发中记录可收回、已转项目记录不可收回/删除；项目创建后进入梗概，各阶段按门禁解锁，全部正文通过后编辑可手动解锁质检，质检通过后项目可完成。

**Doc 协作（核心）**：
1. 四类 Doc 都通过通用编辑器处理，保存完整文档 JSON，后端不依赖其内部结构。
2. 当前稿件与历史版本入口清晰区分：当前稿件只允许当前持有人编辑，历史版本只读回看，不提供版本对比。
3. 共创接力循环成立：作者编辑/提交 → 编辑修改/批注 → 编辑退回或通过 → 作者在退回内容上继续修改 → 再提交，直到通过。
4. 同一 Doc 同一时刻只有一份 `active` CurrentDraft，且只有 `holder_role` 对应角色可保存正文变更。
5. 每次提交、退回、通过都封存 CurrentDraft 并生成 Revision；Revision 永久不可修改，作者和编辑可逐轮回看。
6. 编辑持有编辑权时可直接修改正文，也可添加修订标记、批注和编辑建议；作者持有编辑权时在最新退回稿上继续修改正文。
7. Clean 正文定义为最新稿件或指定 Revision 去除修订标记、批注高亮、编辑建议卡片等协作标记后的原文，不是另一份独立稿件。
8. 工作模式展示全部协作标记，阅读模式隐藏全部协作标记并呈现 Clean 正文。
9. 只有 `已提交待审` 且 `holder_role=editor` 的 Doc 可审核，退回必须保留反馈，审核结果通知作者。

**Release**：页面显示"质检"、内部按 `release` 实现；全部正文通过前不可编辑；编辑手动解锁后初始内容来自已通过正文；修改不回写单章；通过后项目可完成，终稿导出优先取质检 Doc。

**治理与通知**：管理员可管理用户/作者审批/密码/绑定/参数、项目治理（归属/阶段时长/完成/归档/取消/恢复）与终稿下载；关键动作产生通知并可跳转，待我处理聚合当前用户任务。
